# AI Agent 术语表 (Glossary)

本工作区的**官方语言**。所有课程、练习、学习记录都遵循这里的术语。
> 规则：只有当用户**真正理解**一个术语后，才把它收录进来。这是一个「已压缩的知识」记录，不是用来从头读的字典。它会随学习推进持续生长。

## 基础（用户已具备，作为词汇锚点）

**LLM (大语言模型)**:
基于海量文本训练、能根据上文预测下一个 token 的概率模型；是 Agent 的「大脑」。
_别名/避免_：避免与「聊天机器人」混为一谈——聊天只是它的一种用法。

**Prompt (提示词)**:
给 LLM 的输入指令与上下文。一次 LLM 调用 = prompt in → 文本 out。

**Token**:
LLM 处理文本的最小单位（约等于一个字或半个词）。计费、上下文长度都以 token 计。

**Context window (上下文窗口)**:
模型一次能「看见」的 token 上限。超出则需截断或借助外部记忆。

**API call (一次模型调用)**:
向模型发送 prompt、接收输出的无状态请求。**这是「普通 LLM 调用」的基本单元**，也是 Agent 循环里的一步。

## Agent 核心（第 1–2 课确立）

**Agent（智能体）**:
让 LLM 进入一个循环，给它工具与记忆，使其能自主朝目标推进、直到完成的系统。
_别名/避免_：避免把「任何用到 LLM 的程序」都叫 Agent——见 [[#Workflow]]。区分点是「下一步谁决定」。

**Tool use（工具使用 / 函数调用）**:
模型输出一段结构化请求「我想调用工具 X、参数是 Y」；**真正执行该函数的是你的代码**，结果再喂回模型。模型只请求，不执行。

**ReAct**:
最经典的 Agent 循环范式（Yao et al. 2022），由 _Rea_soning + _Act_ing 得名。三步反复：Thought（想）→ Action（做）→ Observation（看结果），直到给出最终答案。

**循环（The Loop）**:
Agent 的本质——一个 `while` 循环：让模型决定下一步 → 若请求工具则执行并把结果喂回 → 否则结束。模型只是循环里的一步。

**Workflow（工作流）**:
步骤由人预先编排、路径固定的 LLM 系统。与 Agent 的区别：下一步由「人定的流程」决定，而非「模型临场决定」。

## 记忆与上下文（第 3 课确立）

**上下文内记忆 (in-context memory)**:
模型此刻在 prompt/messages 里能直接「看见」的记忆。受 [[#上下文窗口]] 硬限制，占 token、贵、易超限。第 2 课的「记忆 = messages」即此类。

**外部记忆 (external memory)**:
存在上下文窗口之外（数据库、向量库、文件），按需检索一小部分塞进窗口的记忆。容量几乎无限，代价是多了「检索」这一步。

**滑动窗口 (sliding window)**:
记忆管理策略——只保留最近 K 条消息、丢最旧的。最省 token，但会丢掉旧信息（对长程依赖任务致命）。

**摘要压缩 (summarization)**:
记忆管理策略——把旧消息压成一条摘要、留最近几条。省空间又保要点，代价是多一次模型调用。

**RAG (检索增强生成)**:
把大量知识存外部（常为向量库），每轮按相关性检索片段塞进上下文。适合大、动态、需溯源的知识库；跨文档综合比长上下文更准（约 +67%）且更省（约 8 倍）。

**Context engineering (上下文工程)**:
优化「填进上下文窗口的整个信息环境」（系统提示、记忆、检索、工具结果），而非只优化问题措辞。被视为 Agent 时代对 prompt engineering 的升级。

**Prompt caching (提示缓存)**:
把稳定前缀（系统提示 + 工具 + 早期历史）缓存复用，按前缀匹配命中；成本 ↓约 90%、延迟 ↓约 85%。不减少 token，是上生产的必杀技。注意：前缀一动，后面缓存全部失效。

## 工具设计（第 4 课确立）

**命名空间化 (namespacing)**:
给相关工具加共同前缀（如 <code>db_query</code> / <code>db_insert</code>）以分组、划定功能边界；既帮模型理解归类，也避免名字冲突。Anthropic 官方推荐做法。

**并行工具调用 (parallel tool calling)**:
模型在一轮响应中请求多个相互独立的工具；运行方需<strong>全部执行</strong>并把所有结果作为一条消息喂回。当工具有依赖关系（B 需要 A 的输出）时必须<strong>串行</strong>。能减少往返轮数、提升性能。

**错误当数据返回 (error-as-data / tool-grounded correction)**:
工具失败时<strong>不抛异常</strong>，而是把错误格式化为结果字符串返回给模型，使其能据此自我纠正（改参数/换工具/如实报告）。需配合最大重试次数防止死循环。属「tool-grounded」纠错（可验证、有效），区别于存疑的「epistemic」纠错。
