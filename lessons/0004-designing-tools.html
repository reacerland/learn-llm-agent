AI Agent 开发与前沿研究 · 第 4 课

工具设计进阶
让模型「愿意调、调得对」

第 2 课你学会了工具的机制（怎么接进循环）。这一课讲设计—— 因为工具是写给一个「读说明书来决定用哪个」的非人类读者（模型）的。设计得好坏， 直接决定模型会不会调、调得对不对。

⏱ 约 18 分钟 🐍 含可运行演示 🛠 错误自愈 + 并行调用 📦 生产必备
本节你会获得
  • 掌握「好工具」的解剖结构：名字、描述、参数各该怎么写。
  • 学会一个反直觉但关键的模式：工具出错要当数据返回，而不是抛异常。
  • 看懂并实现并行工具调用：一轮多工具、一起返回。
  • 知道「工具不是越多越好」——少而精、命名空间化。

01核心认知：工具是写给模型读的

模型怎么决定「这一步用哪个工具」？它读工具的名字和描述，再对照当前任务做判断 （[Anthropic 的 advanced tool use 文档](https://www.anthropic.com/engineering/advanced-tool-use) 明说：工具发现是「按名字和描述匹配」）。

这一句决定一切 工具的描述是影响「调用准确率」最大的因素——比名字、比参数都重要。 官方原话：要写「extremely detailed descriptions（极其详细的描述）」。

所以写工具，本质是在给一个聪明的、但只看说明书的助手写操作手册。

02好工具 vs 差工具：直接对比

同一个「搜索」功能，差定义和好定义天差地别：

❌ 差的定义

{
  "name": "search",
  "description": "搜索",
  "input_schema": {
    "type": "object",
    "properties": { "q": { "type": "string" } }
  }
}

问题：名字太泛（搜什么？）、描述等于没说（什么时候用？返回什么？）、参数 q 含义不明。

✅ 好的定义

{
  "name": "search_web",
  "description": "在公开网络上搜索最新信息。当需要查询训练数据截止日期
    之后的事实、实时数据（股价/天气/新闻）、或模型不确定的内容时使用。
    不要用于写代码或查询内部知识库。返回：最多 5 条结果的标题、摘要与 URL。",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": { "type": "string",
                 "description": "用关键词组成的搜索词，不要用完整问句" }
    },
    "required": ["query"]
  }
}

提炼成四条原则（依据 [Writing effective tools for AI agents — Anthropic](https://www.anthropic.com/engineering/writing-tools-for-agents) 与 [Define tools 官方文档](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools)）：

维度怎么写
名字动词+名词，自解释：search_web / get_weather / send_email。相关工具用命名空间前缀分组：db_query / db_insert。
描述回答三问：什么时候用？做什么？返回什么？必要时写「什么时候【不】用」和边界。
参数名字自解释、每个加 description、标 required；能用 enum 就别用自由文本；给默认值减少必填。
数量少而精。工具太多会撑爆上下文、且模型选错/漏调。按场景分组或做「工具路由」。

03关键模式：错误当数据返回，而不是抛异常

新手最常犯的错：工具一失败就 raise 异常，结果整个 agent 循环崩溃。正确做法——

工具失败时，捕获异常，把它格式化成一段错误字符串作为 tool 结果返回，喂回模型。模型读到错误，就能自我纠正（改参数、换工具、或如实报告失败）。

看演示（code/0004_tool_design_demo.py），这是我亲自跑出的真实输出：

👤 任务: 查一下北京现在几点、天气如何，给我穿搭建议。

── 第 1 轮：模型请求 1 个工具 ──
⚙️  Action  : get_weather({'city': '南京'})
👁  Observe : ERROR: 没有城市「南京」的数据。可选城市：['北京', '上海']
🛠  处理   : 工具返回错误，但循环【不中断】——喂回模型，让它自己改。

── 第 2 轮：模型请求 2 个工具（并行） ──
⚙️  Action  : get_weather({'city': '北京'})
👁  Observe : 5°C，西北风 6 级
⚙️  Action  : get_time({'timezone': '北京'})
👁  Observe : 北京 当前 14:30

✅ Final : 北京现在 14:30，5°C 且大风，建议穿厚外套、注意防风保暖。

关键看 get_weather 的实现——错误是返回值，不是异常：

def get_weather(city):
    if city not in table:
        return f"ERROR: 没有城市「{city}」的数据。可选城市：{list(table)}"  # ← 返回，不是 raise
    return table[city]
两个配套铁律 ① 错误信息要对模型有用——说清「错在哪、可选项」，它才能纠正（光写 "failed" 没用）；
② 一定要配最大重试次数，否则模型可能反复卡在同一个错误上死循环。

04并行工具调用：一轮多个、一起返回

上面的输出里，第 2 轮模型同时请求了天气和时间——这就是并行调用 （[Parallel tool use — Anthropic](https://platform.claude.com/docs/en/agents-and-tools/tool-use/parallel-tool-use)）。

机制很简单，但有两个要点：

  • 你的代码要执行【全部】请求的工具，并把所有结果一起作为一条消息喂回。别只跑第一个。
  • 只对彼此独立的工具并行。如果工具 B 依赖 A 的结果（A 的输出是 B 的输入），就必须串行——模型通常能自己判断，但你在循环里也要意识到这点。
为什么值得用 研究表明（[arXiv](https://arxiv.org/html/2602.07359v1)），用「并行」扩展宽度能提升性能、减少所需轮数——一次拿齐多个独立信息，省掉来回往返。生产里常见一批 5–10 个并行调用。
