01先看清问题：messages 会一直涨

第 2 课的循环里，每一步都往 messages 里 append。对话一长，这条数组就无止境地增长。后果有二：

  1. 成本爆炸：API 按 token 计费，每轮都把全部历史重新发一遍，越往后越贵。
  2. 撑爆窗口：模型有上下文窗口上限（见术语表）。超过就报错，或被迫截断——而截掉的可能正是关键信息。

我把这个抽象问题做成了可运行演示（code/0003_context_management_demo.py），这是我亲自跑出的真实数字：

=== 三种方式 token 对比 ===
  · 原始          16 条消息 ≈ 55 tokens   ← 对话再长，这个数只会涨
  · 滑动窗口       4 条消息 ≈ 15 tokens   ← 最省，但丢了旧信息
  · 摘要压缩       5 条消息 ≈ 43 tokens   ← 省空间，又保住要点
自己跑一遍 python3 code/0003_context_management_demo.py —— 改一改对话内容，观察数字怎么变。眼看比脑想管用十倍。

02关键重分类：两种记忆

第 2 课那句「记忆 = messages」其实只说对了一半——它特指上下文内记忆：

上下文内记忆 (in-context)外部记忆 (external)
存在哪就塞在 messages / prompt 里放在数据库、向量库、文件里
容量受上下文窗口硬限制几乎无限
模型怎么用每次都全部「看见」按需检索一小部分塞进窗口
代价占 token、贵、易超限需要检索（多一步）
一句话 上下文内记忆＝模型此刻「眼前」的东西（贵但即时）；外部记忆＝模型的「笔记本」，要用时翻一页出来。 高级 Agent 的核心本领，就是在两者之间聪明地搬数据。

032026 心智升级：Context Engineering

这一两年的行业共识是：光会写好 prompt 不够了。区别在于——

  • Prompt engineering（提示工程）：优化「怎么问」——把问题措辞写好。
  • Context engineering（上下文工程）：优化「填进窗口的整个信息环境」——系统提示、记忆、检索来的文档、工具结果，怎么编排、何时丢弃。
Prompt 工程优化「问题」；Context 工程优化「问题被回答的条件」。 对 Agent 而言，后者才是主战场。

来源：Effective Context Engineering for AI Agents — Anthropic 及多家 2026 综述。

04三种策略：从简单到强大

策略怎么做优点代价
滑动窗口 只保留最近 K 条，丢最旧的 最简单、最省 token 丢掉旧信息（可能正好是关键的）
摘要压缩 把旧消息压成一条摘要，留最近几条 省空间又保住要点 摘要本身要一次模型调用；可能丢细节
RAG 检索 记忆全存外部，每轮按需检索相关片段 容量无限、可更新、可溯源 工程复杂；检索质量决定一切
选型不是「越高级越好」——够用就行。简单对话用滑动窗口足矣。

这三者还能组合：比如「滑动窗口兜底 + 摘要压缩中期 + RAG 长期」，是生产系统常见架构。

05上生产必杀技：Prompt Caching

它解决的是另一个维度的问题——不减 token，但大幅减钱和时间。原理：

为什么有效 Agent 循环里，系统提示 + 工具定义 + 早期历史在每轮几乎不变。这块「稳定前缀」可以被服务端缓存复用，不必每次重算。
指标效果（Anthropic 官方数据）
成本↓ 约 90%
延迟↓ 约 85%
机制按「前缀匹配」缓存；用 cache_control 标记断点（单次最多 4 个）
三个命中率铁律 ① 系统 prompt 和工具定义尽量保持稳定，别中途改；
② 缓存的是前缀——前面的内容一动，后面全部失效；
③ 因此把不变的东西放前面、易变的放后面（如最新一轮对话）。

06决策表：RAG vs 长上下文

当知识量很大时，「全塞进长上下文」还是「RAG 检索」是高频选择题。实测结论（多家 2026 对比）：

  • 单文档深推理：长上下文更准（约 +34%）。
  • 跨文档综合：RAG 更准（约 +67%），且成本/延迟低约 8 倍。
  • 2026 共识：不是二选一，混合（RAG 取回片段 + 长上下文做深推理）往往是最佳。
场景用哪个
知识库远超窗口（如百万字）RAG
内容频繁更新（商品、新闻）RAG
需要引用来源 / 可溯源RAG
跨多篇文档做综合RAG（+67%）
单文档深推理（代码审查、合同）长上下文（+34%）
既要大规模又要深推理混合

08动手（应用层）

打开 code/0003_context_management_demo.py：

任务 A：在 build_history() 里再加 2 个城市，重跑，观察「原始」和「摘要压缩」的差距是变大还是变小，并用一句话解释为什么。

任务 B：如果你全程只用滑动窗口，这个会话会在哪一步「失忆」？这对什么样的任务是致命的？

A：差距会变大。对话越长，原始 token 线性增长，而「摘要压缩」始终≈「一条摘要 + 最近 K 条」，增长慢得多——这正是摘要策略的价值随对话长度放大。

B：滑动窗口会在「最冷/最暖是哪个城市」这种需要跨历史综合的问题上失忆——因为最早的几个城市数据已被丢弃。对长程依赖（财务对账、多轮订票、客服跟进）类任务是致命的。这正是摘要/RAG 存在的理由。
