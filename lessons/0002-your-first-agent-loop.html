01先纠正一个最大的误区

很多人第一次听说「大模型能调用工具」时，脑子里是这样的画面：

❌ 误以为：「模型调用 get_weather」＝ 模型自己跑去查了天气，把结果算出来了。

真相完全不是这样。真正发生的是两段：

① 请求模型输出一段结构化文字：「我想调用 get_weather，参数 city="北京"」。注意——它只是说要调，自己并没有真的执行。
② 执行你的代码读到这段请求，由你来真正运行 get_weather("北京")，再把结果作为「观察」喂回给模型。
这一刻，黑魔法就消失了 模型不会运行你的函数、不能碰你的数据库、不能联网——它只会说话。 所谓「工具调用」，本质是「模型说了一句你约定好的、机器可解析的话，然后你照着办」。 把这句话想通，后面的一切都顺了。

02Agent 的灵魂：一个 while 循环

把上一课的 ReAct（Thought → Action → Observation）翻译成代码，就是这样一个循环：

def run_agent(user_task):
    messages = [{"role": "user", "content": user_task}]
    while True:                              # ← Agent 的灵魂就是这个循环
        response = llm(messages)             # ① 模型决定下一步
        if response 想调用工具:
            结果 = 你执行那个工具(参数)        # ② 你的代码真正执行
            把结果喂回 messages              # ③ 让模型「看见」结果
        else:                                # ④ 模型不再要工具 → 给最终答案
            return response 的文本

注意三件事：

  • 循环是「你的」，不是模型内部的。是你在 while True 里反复喊模型。
  • 执行工具是「你的」（第 ② 行）。模型只在第 ① 行「请求」。
  • 记忆就是 messages——每一步都把新内容 append 进去，下一轮模型就能看到全部历史（这就是它「记得」的方式）。
为什么必须有个 max_steps while True 有风险：模型可能反复要调工具、永远不停。所以真实代码一定加一个最大步数兜底。这是工程上的常识，面试也会问。

03立刻运行它（零依赖版）

我写了一个不需要任何 API key、不依赖第三方库的最小 Agent。 它的「模型」是个写死的脚本——为的是让你先看清循环本身。完整代码在 code/0002_min_agent_mock.py，关键部分如下：

# 工具：真正「做事」的 Python 函数（你的代码）
def get_weather(city: str) -> str:
    return {"北京": "5°C，西北风 6 级"}.get(city, "暂无数据")

TOOLS = {"get_weather": get_weather}     # 工具名 -> 函数

def run_agent(user_task, max_steps=5):
    messages = [{"role": "user", "content": user_task}]
    for step in range(max_steps):        # ← 循环（带兜底）
        response = llm(messages)         # ① 模型决定下一步
        if response["kind"] == "tool_use":
            name, args = response["name"], response["input"]
            result = TOOLS[name](**args) # ② 你的代码真正执行
            messages.append({"role": "user", "content": f"{name} => {result}"})  # ③ 喂回去
        else:
            return response["text"]      # ④ 给最终答案

在项目根目录运行（任何装了 Python 的机器都行）：

python3 code/0002_min_agent_mock.py

你会看到——这是我亲自跑出来的真实输出，不是编的：

👤 任务: 今天北京适合穿什么？

🧠 Thought : 我需要调用 get_weather({'city': '北京'})
⚙️  Action  : get_weather(**{'city': '北京'})
👁  Observe : 5°C，西北风 6 级

✅ Final   : 今天北京 5°C 且大风，建议穿厚外套、围巾，注意防风保暖。
对照第 1 课看 Thought / Action / Observe / Final 正是上一课 ReAct 的四行。 心智模型 ↔ 代码，一一对应。现在你亲手把它们连起来了。

04换上真实模型，只改一个函数

当你有了 API key，要做的事少得惊人——只把那个假的 llm() 换成对 Claude 的真实调用，循环主体一行都不用改。

真实版的工具「说明书」是这样的（模型靠它判断何时、怎么调）：

TOOL_SPECS = [{
    "name": "get_weather",
    "description": "获取指定城市的实时天气。",
    "input_schema": {
        "type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"],
    },
}]

真实版的完整代码在 code/0002_min_agent_anthropic.py。 对比两个文件你会发现：结构完全一样，只有 llm() 内部不同。这就是 Agent 的本质——一个与具体模型解耦的循环。

06动手改造（应用层练习）

打开 code/0002_min_agent_mock.py，做这两件事——动手改是建立 storage strength 最有效的方式：

任务 A：加一个 calculate(expression) 工具（用 Python 内置的 eval 即可），再让脚本先问天气、再算一笔账，跑通两个工具的循环。

任务 B：把 max_steps 改成 1，观察会发生什么，并用一句话解释原因。

把 max_steps=1 后，循环只允许跑一轮。第一轮模型请求调用工具，工具执行完、结果喂回去——但循环已到上限，没机会让模型看到结果并给出最终答案，于是打印「⚠️ 达到最大步数，强制停止」。

这恰好证明了：Agent 需要「至少 N 步」才能完成一个需要 N 次工具调用的任务。max_steps 不是摆设，它是任务复杂度的硬约束。
